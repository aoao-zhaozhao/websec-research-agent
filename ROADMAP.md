# My Agent — 生产级开发路线图

## 当前状态: v0.5

```
已具备: LangGraph 引擎 + 9 个工具 + RAG 知识库 + Web 前端
能做的: 输入 URL → 自动爬取 + 扫描 → search_knowledge 查知识库 → 输出带 CVE/CVSS/修复方案的报告
```

---

## v0.5 — RAG 知识库 ✅

> 从"模型自己判断"升级为"有据可查"

### 工程改动

| 模块 | 内容 |
|---|---|
| 文件拆分 | `agent/core.py` → `agent/` `config.py` `prompts.py` `agent.py` `tools/` `rag.py` |
| 知识库 | `agent/knowledge/` — OWASP Top 10、CVE 精选、修复方案 Markdown |
| 向量库 | Chroma 持久化 + Qwen3-Embedding-0.6B（1024 维本地模型） |
| 精排 | Qwen3-Reranker-0.6B CrossEncoder 两阶段检索 |
| 新工具 | `search_knowledge(query)` — 两阶段检索最相关漏洞案例和修复建议 |
| System Prompt | "扫描发现可疑行为时，先调用 search_knowledge 查找已知漏洞模式，再下结论" |

### 实际交付

```
v0.4: 8 个工具，~900 行 core.py
v0.5: 9 个工具，14 个文件，~1500 行
      两阶段 RAG: Embedding 粗排 → Reranker 精排
      知识库: 3 个源文档 → 29 个向量块
      "发现 SQL 注入，匹配 CVE-2021-44228 (Log4Shell)，CVSS 10.0，
       修复: 参数化查询..."   ← 有 CVE 编号 + CVSS + 代码级修复方案
```

## v0.6 — 认证扫描

> 能扫登录后的页面（大部分漏洞藏在这）

### 工程改动

| 模块 | 内容 |
|---|---|
| 会话管理 | Agent 自动处理登录表单 → 维持 Cookie/Token → 带认证状态继续扫描 |
| 新工具 | `login(url, username, password)` — 自动识别表单字段并发起登录 |
|  | `set_auth_header(key, value)` — 手动设置 Bearer Token / API Key |
|  | `current_auth_state()` — 查看当前认证状态 |
| 安全 | 凭证仅存内存，不回显，不记录日志 |

### 能力提升

```
v0.5: 只能扫公开页面
v0.6: 登录后台 → 带着 Cookie 扫管理面板 → 发现越权漏洞/未授权访问
```

---

## v0.7 — 持久化 + 专业报告

> 扫描结果入库，能查历史，能出 PDF

### 工程改动

| 模块 | 内容 |
|---|---|
| 数据库 | SQLite（零配置）→ SQLAlchemy ORM → 3 张表: scans / vulnerabilities / pages |
| 存储 | 每次扫描结果、漏洞列表、页面快照落库 |
| 历史 | `/api/scans` — 历史扫描列表、同站多次扫描对比 |
| 报告 | Markdown → PDF 导出（`weasyprint` 或 `reportlab`） |
| 新工具 | `compare_scan(id1, id2)` — 对比两次扫描，看漏洞是否修复 |

### 能力提升

```
v0.6: 扫完就没了，关浏览器 = 结果丢失
v0.7: 扫描落库 → 随时回看 → 周报/月报导出 → "上次那 3 个高危修了吗？" → 重新扫对比
```

## v0.8 — 扫描引擎深化

> 不只扫"有没有"，而是扫"有多深"

### 工程改动

| 模块 | 内容 |
|---|---|
| Payload 库 | `agent/payloads/` — SQLi/XSS/SSRF/XXE/SSTI/命令注入 各 20+ payload |
| 探测策略 | 时间盲注（sleep）、OOB 回连检测、WAF 绕过变体 |
| 并发扫描 | `asyncio.gather` — 多页面并行检测，不排队 |
| 新工具 | `fuzz_url(url, param, payload_type)` — 对指定参数批量注入 |
|  | `detect_tech_stack(url)` — 识别框架/语言/服务器版本（为精准 exploit 做准备） |
| 误报抑制 | 同样 payload 发两次，结果不一致 → 标记"可能误报" |

### 能力提升

```
v0.7: 每个点测 2-3 个 payload
v0.8: SQLi 含时间盲注/WAF绕过/二次编码，XSS 含 DOM型/存储型探针
      自动识别 "ThinkPHP 5.0" → 针对性 payload
```

---

## v0.9 — 任务队列 + 安全加固

> 从"单用户玩具"变成"能给人用的服务"

### 工程改动

| 模块 | 内容 |
|---|---|
| 任务队列 | `asyncio.Queue` — 多用户提交扫描排队，限制并发 |
| 速率控制 | 对同一目标限速（如每秒 3 请求），避免打挂靶场或被 WAF 封 |
| 鉴权 | 简易 API Key 鉴权（`X-API-Key` header） |
| 沙箱隔离 | 扫描在独立子进程中执行，超时自动 kill |
| 日志 | 结构化日志（JSON 格式）+ 审计日志（谁什么时候扫了谁） |
| 配置 | `.env` 支持扫描超时、并发数、速率限制等 |

### 能力提升

```
v0.8: 一个人用，崩了重启
v0.9: 多人排队扫描 → 速率限制保护目标 → 出问题有日志可查
```

## v1.0 — 多租户 + 部署

> 完整产品形态

### 工程改动

| 模块 | 内容 |
|---|---|
| 数据库升级 | SQLite → PostgreSQL（生产环境） |
| 多租户 | 每个用户独立的数据视图，权限隔离 |
| Docker | `Dockerfile` + `docker-compose.yml` — 一键部署 |
| CI/CD | GitHub Actions — 提交自动测试 + 构建镜像 |
| 前端完善 | React/Vue 重构前端（当前 `index.html` 撑不住了） |
| 监控 | Prometheus metrics 端点（扫描次数、漏洞数、响应时间） |
| 文档 | 部署文档 + API 文档 + 用户手册 |

## 不变的部分

| 层 | 说明 |
|---|---|
| FastAPI 控制面 | 从一开始就对了，v0.1 → v1.0 不会大改 |
| LangGraph 引擎 | v0.3 选型正确，后续只加工具不加引擎复杂度 |
| 分层解耦 | Agent 不 import FastAPI，永远如此 |
