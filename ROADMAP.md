# My Agent 路线图

## 产品定位

My Agent 的产品定位仍是 Web 安全研究与授权测试 Agent，不是面向用户的通用 CTF
解题器。CTF 题目在本项目中的角色是标准化、可判定、可复现的能力评测集：它们用来
检验 Agent 是否真的具备观察、推理、写脚本、执行、修正和验证的闭环能力。

因此，能力建设应遵循以下原则：

- WebSec 工作流是生产主路径，现有 Web 扫描、验证、证据和报告能力继续演进。
- CTF benchmark 是离线评测路径，不改变生产 Agent 的授权范围或产品交互。
- 仅当 benchmark 持续暴露某项能力缺口时，才为生产 Agent 增加相应的、受限的工具能力。
- 不为“覆盖所有 CTF 类别”而盲目接入重型 Reverse/Pwn 工具链。

目标闭环是：

```text
输入或授权目标 -> 观察 -> 提出假设 -> 写脚本/调用工具 -> 执行
               -> 读取证据 -> 修正 -> 验证结果 -> 记录指标
```

任何文件修改、编译或命令执行都必须在隔离环境中完成，Agent 不获得宿主机的自由访问能力。

## 当前基线

当前版本：v1.7 WebSec Research Agent。

- 使用 LangGraph 编排，并提供 FastAPI/WebSocket 工作台。
- 提供 39 个 Web 安全、证据、Skill 和案例记忆工具。
- 支持结构化 `ToolResult` 和扫描生命周期事件。
- 覆盖侦察、注入验证、SSRF、JWT、授权、OOB 确认和 CSS 外带等 Web 工作流。
- 具备 RAG 检索、案例记忆和受治理的 Skill 生命周期。
- 已记录运行、工具行动、协议失败、模型 token 用量和运行指标，并在工作台展示。

当前限制：

- 没有可隔离运行 Python、编译验证代码、分析附件或调用常见 CLI 的执行环境。
- 没有脚本、下载文件、解码结果和生成产物的工作区管理。
- 对话历史仍只保存在浏览器 LocalStorage；服务端没有可跨浏览器、重启后恢复的会话记录。
- 本地会话最多保留 30 条，移动端隐藏侧栏后也没有会话历史入口。
- 当前提示词偏线性扫描，不能稳定驱动“观察 - 尝试 - 修正”的实验循环。

## 目标架构

```text
生产请求：授权 Web 安全研究             评测请求：本地 CTF benchmark
               |                                  |
               +----------- 统一任务契约 ----------+
                                  |
                                  v
                    规划、证据、行动与预算账本
                                  |
                                  v
                  隔离分析执行器和受限工作区工具
                                  |
                                  v
             WebSec 证据/报告              本地 judge 验证结果
                                  |
                                  v
                 指标、回放、案例记忆与 Skill 治理
```

CTF 评测题只使用本地附件、固定服务和本地 judge；不会自动把外部 CTF 平台或任意
互联网目标纳入生产 Agent 的操作范围。

## v1.7：度量与可靠性基础

目标：先让当前 Agent 的运行数据准确、可复算，再以数据决定后续工具投入。

### 交付内容

- 每次工具调用只记录一条终态观测，修复重复写入。
- 缺失或无效的 `ToolResult` 记录为带原因的协议失败，不再静默记为 `unknown`。
- 建立任务/运行账本：任务类型、运行模式（生产或评测）、授权范围、开始/结束时间、
  预算、最终状态和证据引用。
- 建立行动账本：有序工具调用、脱敏后的输入、状态、错误类型、耗时、stdout/stderr
  引用和产物关联。
- 每次模型请求记录模型名、输入/输出 token、缓存 token（若供应商提供）和成本。
- 建立评测记录：fixture 版本、本地 judge、候选结果指纹、验证结果和失败原因。
- 在 API 和工作台展示总体、按 Web 能力域和按 benchmark 类别的指标。

### 指标定义

| 指标 | 定义 |
|---|---|
| Solve rate | 本地 judge 确认通过的 benchmark 任务数 / 已完成 benchmark 任务数 |
| 首次有效行动比例 | 首次行动产生任务预定义有效观察的任务数 / 已尝试任务数 |
| 工具失败率 | 以 `error` 结束的工具调用数 / 已完成工具调用数；协议失败单独统计 |
| Token 成本 | 供应商返回的 usage 字段乘以配置的模型价格表后求和 |
| 预算耗尽率 | 因 token、工具次数、CPU、墙钟时间或网络预算停止的运行数 / 已尝试运行数 |

Web finding、模型自述或看起来像 flag 的字符串都不是 solve。评测成功只能由本地 judge
确认；生产任务则以结构化证据和报告质量衡量，不应伪装成 CTF 解题结果。

### 发布门槛

- 集成测试中不再产生重复终态行动记录。
- 每个已注册工具都返回有效 envelope 或分类明确的协议失败。
- 指标可从持久化运行数据稳定复算。
- 现有 WebSec 回归测试保持通过。

## v1.7.1：服务端会话持久化

目标：将“用户会话”和“运行遥测”统一为可恢复的历史记录，避免刷新、重启、换浏览器或
移动端布局导致历史不可见或丢失。

### 数据模型

- 在 `telemetry.db` 增加 `conversations`，保存会话 ID、标题、创建/更新时间和状态。
- 为 `telemetry_runs` 增加 `conversation_id`，将用户的一次扫描或提问关联到会话。
- 复用已有 `telemetry_actions`、`telemetry_model_usage` 和 `telemetry_evaluations`，不再复制
  工具轨迹、token 用量或判题记录。
- 用户消息来自运行输入；助手最终答复来自运行摘要；运行中的部分输出按可恢复状态持久化。

### API 与前端

- 新增会话列表、创建、读取、重命名和删除 API。
- WebSocket 消息必须携带 `conversation_id`，服务端先写入用户输入，再更新运行和最终答复。
- 前端以服务端会话为真实来源，LocalStorage 仅作短期缓存。
- 首次升级时，仍存在于 LocalStorage 的旧会话可一次性、幂等地导入服务端。
- 桌面端侧栏和移动端独立入口都可访问同一份会话列表；历史运行作为会话下的可展开审计项。
- “清空历史”必须明确确认，并同步删除服务端记录，不再只清除浏览器状态。

### 安全与发布门槛

- 入库前脱敏 Cookie、Bearer token、API key、密码和其他凭据字段。
- 重启服务、刷新页面、使用新的浏览器 profile 后，已保存会话仍可恢复。
- 删除会话后，其关联运行、行动和评测记录按明确的保留策略处理。
- 回归测试覆盖创建、恢复、重命名、删除、LocalStorage 导入和凭据脱敏。

## v1.7.2：大响应定向检索

目标：让 Agent 能处理大于模型摘要窗口的已授权 HTTP 响应和渲染页面，在已知线索存在时
定位证据，而不把数十 KB 原文直接注入模型上下文。

### 交付内容

- 新增 `search_http_body(url, keyword_or_regex)`：工具完整读取受限的同源文本响应，只返回响应长度、
  内容哈希、命中次数、偏移量和受限上下文。
- 新增 `search_rendered_dom(url, keyword_or_regex)`：对 Playwright 已渲染 DOM 的完整文本执行同样检索。
- 结果协议区分“没有命中”“响应截断”“不支持的内容类型”“超出响应上限”和“正则错误”。
- 用户目标或 benchmark 任务明确要求某个关键词/flag 模式，且响应长度超过展示摘要时，规划器优先调用
  定向检索，而不是继续进行无关 payload 猜测。
- benchmark 模式通过候选 flag 正则或本地 judge 验证后才允许结束；生产 WebSec 模式仍以漏洞证据和报告为终点。

### 发布门槛

- 75KB 以上的 fixture 将目标字符串置于 60KB 后，Agent 必须通过定向检索定位它。
- 单次工具输出受限，测试确认完整响应不会被注入模型上下文或持久化到案例记忆。
- 同源、内容类型、响应体上限、超时和正则解析错误均有回归测试。

## v1.8：隔离分析执行能力

目标：为 Web 安全研究和离线 benchmark 提供受限、可观察的脚本和工具执行能力。

### 沙箱模型

- 每个任务使用版本固定、短生命周期的容器或虚拟机。
- `/input` 为只读输入，`/work` 可写，`/output` 是显式导出的结果目录。
- 不挂载宿主文件系统、Docker socket、凭据目录或项目仓库；执行用户为非 root。
- 限制 capabilities、进程数、CPU、内存、磁盘、墙钟时间和 stdout/stderr 大小。
- 默认拒绝网络；生产 Web 任务只能访问任务契约中已授权的目标，评测只访问固定本地服务。
- 任务结束销毁沙箱，保留行动记录和声明的输出产物供回放。

### 初始工具集

- `workspace_list`、`workspace_read`、`workspace_write`、`workspace_search`。
- `run_command(argv, cwd, timeout, output_limit)`，使用结构化参数执行常见 CLI。
- `compile_source(language, sources, output_name, flags)`，用于受控验证程序和 PoC 编译。
- `inspect_file`、`inspect_archive`、`extract_archive` 和受限的文本/二进制解码。
- `artifact_register`、`artifact_diff`，追踪脚本、二进制和关键输出。

执行结果必须包括退出码、耗时、资源摘要、受限 stdout/stderr、产物引用和分类错误；不得将
无限长终端输出直接注入模型上下文。

### Agent 工作流

将扫描器专用的线性流程升级为证据驱动状态机：

```text
triage -> observe -> hypothesize -> execute -> evaluate -> iterate
       -> verify -> report
```

每轮都要提出可证伪的假设，并引用上一轮支持该假设的证据。达到任务验证条件、预算耗尽，
或没有能检验新假设的行动时停止。

### 发布门槛

- 回归任务要求 Agent 写入 Python 脚本，观察一次可控失败，修补后得到预期输出。
- 覆盖超时、内存限制、进程限制和网络阻断的集成测试。
- 沙箱逃逸测试无法读取或修改宿主文件。
- 在新建沙箱中可回放已保存的执行产物。

## v1.9：CTF Benchmark 与能力映射

目标：用小而稳定的 CTF 风格任务集测试能力，而不是把 CTF 做成产品功能。

### Benchmark 设计

每个 fixture 都必须固定：题目版本、输入文件或本地服务、允许网络范围、预期结果、本地
judge、时间预算、工具预算和 token 预算。至少覆盖以下能力域：

| 能力域 | 代表任务 | 对生产 WebSec 的价值 |
|---|---|---|
| Web 观察与验证 | 参数、会话、响应差异、Web 附件 | 验证现有扫描与证据能力 |
| 脚本迭代 | 写 Python、读取异常、修复重跑 | 支撑定制分析与验证脚本 |
| 文件与压缩包 | 格式识别、嵌套解包、内容提取 | 处理下载文件、备份、日志和泄露附件 |
| 解码与数据处理 | Base64、URL、压缩、结构化数据 | 支撑 payload、token、日志与响应分析 |
| 编译验证 | 小型验证程序、PoC 构建 | 验证修复假设或受控复现 |

Reverse 或 Pwn fixture 可以作为后续诊断样本，但不构成必须交付的生产 profile。

### 评测输出

- 同时按 benchmark 类别和底层能力域统计 solve rate、首次有效行动比例、工具失败率、
  中位耗时和 token 成本。
- 区分模型决策失败、工具失败、输入/fixture 故障和预算耗尽。
- 每次评测运行保留可回放行动序列和产物哈希。
- 评测结果必须能回答“哪项生产能力不足”，而不只是“哪道 CTF 没做出来”。

### 发布门槛

- 每个能力域包含正例、反例和资源限制样例。
- 每个 solve 都由本地 judge 验证。
- 固定模型、镜像和预算下，新版本不会让已有能力域的结果回退。

## v2.0：基于缺口的能力扩展

目标：只补 benchmark 已证明会影响生产 WebSec 质量的能力缺口。

候选方向按评测结果排序，不预先承诺全部实现：

- 更强的文件和二进制静态分析，用于泄露附件、备份文件和下载产物。
- 可复用的数据提取、解码、结构化解析和差异比较工具。
- 用于授权复现的受控编译、测试和协议分析能力。
- 对现有 Web 工具的策略改进，例如基于证据选择下一步验证，而非固定扫描顺序。
- 只有当实际 Web 安全任务和 benchmark 都证明必要时，才引入特定逆向、调试或二进制工具。

每一项能力扩展都必须先定义：生产场景、隔离边界、结构化结果、fixture、错误分类和成功指标。

## v2.1：长期评测、回放与学习

目标：根据可验证证据持续提升，而不是根据单次成功案例扩张工具集。

- 建立版本固定的 benchmark 仓库，记录 fixture、judge、镜像 digest、模型配置和预算。
- 可从任务契约、行动序列和产物哈希完整回放一次运行。
- 按版本比较 Web 能力域和 benchmark 能力域的指标，识别提升、回退和成本变化。
- 将已验证的生产案例和 benchmark 成功路径沉淀为案例记忆。
- 只有多个独立案例证明可迁移模式后，才将经验提升为可复用 Skill。
- 每次发布前运行历史 solved 集和失败回归集。

## 建议实施顺序

1. 完成 v1.7：遥测 schema、重复事件修复和指标 API。
2. 完成 v1.8：先实现单一 Python 沙箱镜像、工作区工具和执行产物记录。
3. 增加任务契约、预算、证据驱动状态机和本地 judge 接口。
4. 完成 v1.9：建立覆盖脚本、文件、解码和 Web 验证能力的本地 benchmark。
5. 依据 benchmark 与真实 WebSec 任务共同暴露的缺口实施 v2.0 工具扩展。
6. 用 v2.1 的长期回归和成本数据决定后续投入，不按 CTF 类别堆砌功能。

## 目标目录结构

```text
agent/
  execution/
    runner.py            # 沙箱生命周期、资源限制和网络策略
    images.py            # 版本固定的执行镜像定义
    workspace.py         # 受限工作区与产物导出
    artifacts.py         # 产物收集、哈希和回放信息
  runtime/
    task.py              # 生产/评测统一任务契约
    workflow.py          # 证据驱动的迭代状态机
    judge.py             # 本地 benchmark judge 接口
  telemetry/
    store.py             # 运行、行动、usage 和评测记录
    metrics.py           # 能力域和版本聚合指标
  tools/
    workspace_tools.py
    execution_tools.py
    artifact_tools.py
tests/
  fixtures/benchmark/
  integration/execution/
  evaluation/
```

## 非目标

- 不将 My Agent 改造成面向用户的通用 CTF 解题器。
- 不向模型暴露不受限制的宿主机 Shell。
- 不将任意互联网目标视为已授权的测试服务。
- 不将模型文字、flag 样式字符串或单个 finding 当作已验证成功。
- 没有生产场景、隔离边界、fixture 和结构化结果协议时，不盲目增加大量 CLI 工具。
